# Panda Trace collector labels

Panda Agent can optionally add Docker labels that let a host-level `panda_trace_collector` discover selected Panda containers and forward their logs to Panda Trace.

This is a labels-only integration:

- disabled by default;
- no collector container is added to the Panda Agent stack;
- only explicitly selected services are labeled;
- app containers must not receive `PANDA_TRACE_KEY` or other Trace read/admin credentials.

## Why the collector stays outside the stack

`panda_trace_collector` follows Docker container logs, so it needs Docker CLI/socket access. Keeping it operator-owned avoids adding Docker socket access to the Panda Agent compose stack and avoids coupling Panda Agent to panda-trace packaging details.

Install and run the collector from the `panda-trace` project, for example with its systemd installer/runbook. Put collector settings in the collector's own host env file such as `/etc/panda-trace-collector.env`, not in Panda Agent `.env`.

## Configure labels in Panda Agent

In Panda Agent `.env`:

```env
PANDA_TRACE_COLLECTOR_ENABLED=true
PANDA_TRACE_COLLECTOR_SERVICES=core,telegram,discord,environment-manager,runners,wiki,caddy
PANDA_TRACE_ENVIRONMENT=prod
PANDA_TRACE_SOURCE_CORE=src_...
PANDA_TRACE_SOURCE_TELEGRAM=src_...
PANDA_TRACE_SOURCE_DISCORD=src_...
PANDA_TRACE_SOURCE_ENVIRONMENT_MANAGER=src_...
PANDA_TRACE_SOURCE_RUNNERS=src_...
PANDA_TRACE_SOURCE_WIKI=src_...
PANDA_TRACE_SOURCE_CADDY=src_...
```

Supported service selectors:

| Selector | Compose service label value |
| --- | --- |
| `core` | `panda-core` |
| `browser` | `panda-browser-runner` |
| `telegram` | `panda-telegram` |
| `discord` | `panda-discord` |
| `whatsapp` | `panda-whatsapp` |
| `gateway` | `panda-gateway` |

When enabled, every selected service must have its matching `PANDA_TRACE_SOURCE_*` value. The generated compose overlay adds exactly these labels:

```yaml
panda_trace.enabled: "true"
panda_trace.source_id: "src_..."
panda_trace.service: "panda-core"
panda_trace.environment: "prod"
```

`PANDA_TRACE_ENVIRONMENT` defaults to `prod` when omitted or blank.

## Collector credentials

Do **not** set `PANDA_TRACE_KEY` in Panda Agent `.env`. The docker stack helper rejects it unconditionally so it cannot be passed into app containers through compose env files.

Set collector credentials only in the host-level collector env, for example:

```env
PANDA_TRACE_BASE_URL=https://trace.example.com
PANDA_TRACE_KEY=ptr_... # dedicated logs:write key scoped to the selected source(s)
PANDA_TRACE_BATCH_SIZE=100
PANDA_TRACE_FLUSH_SECONDS=5
PANDA_TRACE_DISCOVERY_INTERVAL_SECONDS=30
PANDA_TRACE_DEFAULT_SEVERITY=info
PANDA_TRACE_DOCKER_BIN=/usr/bin/docker
```

Use a dedicated `logs:write` key scoped to the selected source IDs. Do not use an owner/admin/read key for log collection.

## Apply and verify

After editing `.env`, run:

```bash
./scripts/docker-stack.sh up
```

To inspect the generated labels without printing secrets:

```bash
grep -n "panda_trace" .generated/docker-compose.remote-bash.external-db.runners.yml
```

On the host, verify the collector can discover only the selected containers:

```bash
docker ps --filter label=panda_trace.enabled=true
```

## Troubleshooting

- `PANDA_TRACE_SOURCE_<SERVICE> is required`: the service is listed in `PANDA_TRACE_COLLECTOR_SERVICES` but its source ID is missing.
- `PANDA_TRACE_KEY must not be set`: move the key out of Panda Agent `.env` and into the collector's host env file.
- No logs arrive: confirm the selected container is running, has `panda_trace.enabled=true`, the collector has Docker access, and the collector key has `logs:write` permission for the labeled source ID.
- Gateway labels require the gateway service to be generated by the docker stack (`PANDA_GATEWAY_ENABLED=true` or public gateway config).
